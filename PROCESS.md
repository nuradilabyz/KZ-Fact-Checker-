# 🛡 KZ Fact-Checker — Подробное описание проекта

## Краткое описание

Автоматизированная система фактчекинга казахскоязычного контента из социальных сетей (Threads). Система ежедневно сканирует посты казахских новостных аккаунтов, извлекает из них проверяемые утверждения и сопоставляет их с базой знаний из статей **Factcheck.kz** с помощью векторного поиска (RAG) и языковой модели (GPT).

---

## Стек технологий

| Компонент | Технология |
|---|---|
| Оркестрация | Apache Airflow |
| Бэкенд API | FastAPI |
| Фронтенд | Streamlit |
| База данных | PostgreSQL + pgvector |
| Эмбеддинги | OpenAI `text-embedding-3-small` |

| Скрапинг статей | Requests + BeautifulSoup + Wayback Machine |
| Инфраструктура | Docker / Docker Compose |

## Пошаговый процесс разработки

### Шаг 1 — Постановка задачи и архитектура

Цель: создать систему, которая:
1. Собирает посты из Threads (казахские новостные аккаунты)
2. Извлекает из каждого поста конкретные проверяемые утверждения (факты)
3. Сравнивает их с базой статей factcheck.kz
4. Выносит вердикт: **SUPPORTED / REFUTED / NOT_ENOUGH_INFO**
5. Показывает результаты в удобном веб-интерфейсе

Была выбрана архитектура **RAG (Retrieval-Augmented Generation)**:
- Статьи Factcheck.kz разбиваются на чанки → переводятся в векторы (embeddings) → хранятся в PostgreSQL с расширением `pgvector`
- При проверке факта: факт тоже превращается в вектор → находятся похожие чанки → GPT анализирует, подтверждают ли они факт

---

### Шаг 2 — Настройка Docker-инфраструктуры

Создан `docker-compose.yml` с 5 сервисами:

```
postgres     — PostgreSQL + pgvector (хранение статей, чанков, результатов)
airflow-init — Инициализация базы данных Airflow
airflow-scheduler — Планировщик задач Airflow
api          — FastAPI сервер (RAG логика)
streamlit    — Веб-интерфейс
```

- Написаны `Dockerfile.api` и `Dockerfile.ui` для сборки образов
- Создан `db/init.sh` — инициализационный скрипт базы данных, создающий таблицы: `fact_articles`, `fact_chunks`, `matches`, `threads_posts`, `threads_claims`
- Файл `.env` содержит все переменные окружения: ключи OpenAI, параметры БД, настройки скрапера

---

### Шаг 3 — Скрапинг и индексация Factcheck.kz

**Файл:** `ingestion/factcheck_scraper.py`

Задача: собрать все ~1319 казахскоязычных статей с Factcheck.kz и сохранить их в векторную базу данных.

**Проблема:** сайт защищён Cloudflare — прямые запросы блокировались.

**Решение:** использование Wayback Machine (web.archive.org):
1. CDX API Wayback Machine возвращает список всех архивных URL для `factcheck.kz/kaz/*`
2. Для каждого URL скачивается архивная версия страницы
3. HTML парсится с помощью BeautifulSoup: извлекается заголовок, текст, дата публикации, вердикт
4. Текст делится на чанки по ~500 символов с перекрытием
5. Каждый чанк отправляется в OpenAI для получения embedding-вектора
6. Вектор сохраняется в PostgreSQL (таблица `fact_chunks`) с использованием pgvector

**Детали реализации:**
- Дедупликация через `content_hash` — уже обработанные статьи пропускаются
- Задача упала с таймаутом через 2 часа на 191-й статье → увеличен `execution_timeout` в DAG до 6 часов
- Итого проиндексировано: **260 статей → 2,757 чанков**

**Airflow DAG:** `factcheck_ingest` — запускается каждый час, подбирает новые/изменённые статьи

### Шаг 4 — RAG API (FastAPI)

**Файл:** `api/main.py`, `api/prompt.py`, `api/embedder.py`

Создан REST API с эндпоинтами:

#### `POST /check` — Проверка факта
1. Принимает текст утверждения
2. Создаёт его embedding через OpenAI
3. Выполняет косинусный поиск по `fact_chunks` через pgvector
4. Топ-N похожих чанков передаются в GPT как контекст
5. GPT возвращает: `verdict`, `confidence`, `explanation_kk`, `best_match`

#### `POST /extract_claims` — Извлечение фактов из поста
1. Принимает текст поста из Threads
2. GPT анализирует текст и извлекает конкретные проверяемые утверждения
3. Возвращает список `claims_kk` (на казахском), `topics`, `lang_detected`

#### `GET /top_news` — Топ новостей из Threads
- Возвращает последние проверенные посты из базы данных

### Шаг 5 — Интеграция Threads (долгий путь)

Это был самый сложный этап, прошедший через несколько итераций:

#### ❌ Попытка 1 — Apify (не сработало)
- Актор `apify/instagram-threads-scraper` оказался платным
- Другие акторы (`usksato~threads-scraper`, `dacoder/threads-search-post-scraper`) не существовали или не работали
- `apify/playwright-scraper` требовал активный Apify-токен → отказались

#### ❌ Попытка 2 — Прямой парсинг Threads HTML (не сработало)
- Threads требует авторизацию, без куки возвращается страница логина
- GraphQL API Threads приватный и требует сессии

#### ❌ Попытка 3 — RapidAPI `threads-scraper` (не сработало)
- Асинхронный API, требующий polling — нестабильный

#### ⚠️ Попытка 4 — RapidAPI `threads-api4` (частично сработало)
- Найден правильный двухшаговый флоу:
  1. `GET /api/user/info?username=tengrinewskz` → получаем `user_id`
  2. `GET /api/user/posts?user_id=63374175859` → получаем посты
- **Проблема с юзернеймами:** `tengrinews_kz` не существует, правильно: `tengrinewskz`
- Найдены и захардкожены правильные ID аккаунтов:
  - `tengrinewskz` → `63374175859`
  - `informburo.kz` → `63390508341`
  - `azattyq` → `63274518510`
  - `factcheck.kz` → `76872015773`
- **Проблема:** бесплатный тариф RapidAPI — всего **50 запросов в месяц** → исчерпали в ходе тестирования

#### ✅ Попытка 5 — Playwright + личный аккаунт через `sessionid` куки
- Пользователь предоставил `sessionid` куки из своего браузера
- Реализован скрипт `ingestion/threads_topnews.py` с Playwright:
  - Куки инжектируются в браузерный контекст Playwright
  - Открывается профиль каждого аккаунта: `https://www.threads.com/@tengrinewskz`
  - Извлекаются тексты постов из DOM: `div[data-pressable-container="true"]`
  - Тексты очищаются от шума (лайки, репосты, кнопки UI)
  - Посты сохраняются в `threads_posts`, утверждения в `threads_claims`, результаты в `matches`

### Шаг 6 — Airflow DAGs

Созданы 2 DAG'а:

#### `factcheck_ingest` (каждый час)
- Обнаруживает новые статьи Factcheck.kz через CDX API
- Парсит, чанкует, создаёт embeddings → сохраняет в PostgreSQL

#### `threads_nightly_topnews` (каждую ночь)
- Запускает `threads_topnews.py`
- Для каждого аккаунта: Playwright берёт топ-20 постов через сессию
- Каждый пост → `extract_claims` → список фактов
- Каждый факт → `check` → вердикт → сохранение в `matches`

---

### Шаг 7 — Streamlit UI

**Файл:** `ui/app.py`

Создан красивый тёмный веб-интерфейс с двумя вкладками:

#### 1️⃣ Посттан фактілерді табу (Extract Facts)
- Пользователь вставляет любой текст (пост из Threads)
- Система автоматически:
  1. Извлекает проверяемые утверждения (через `/extract_claims`)
  2. Проверяет каждое утверждение (через `/check`)
  3. Показывает для каждого факта:
     - Цветной бейдж вердикта (✅ РАСТАЛДЫ / ❌ ЖАЛҒАН / ⚠️ АҚПАРАТ ЖЕТКІЛІКСІЗ)
     - Полоску уверенности (Confidence bar)
     - Ссылку на совпавшую статью Factcheck.kz
     - Объяснение на казахском
     - Раскрывающийся блок с цитатами из источника

#### 2️⃣ Threads: Топ Жаңалықтар (Top News)
- Показывает результаты ночной проверки постов из Threads
- Для каждого поста: вердикт, уверенность, объяснение, ссылка на статью

**Исправления интерфейса:**
- Удалена дублирующая вкладка "Сезікті мәтінді тексеру"
- Удалено поле "Post ID" (лишнее)
- Исправлен параметр `similarity_threshold` — снижен с `0.75` до `0.5` (казахский текст получает косинусное сходство ~0.67, более высокий порог блокировал все совпадения)

---

### Шаг 8 — Отладка и стабилизация

Основные проблемы и их решения:

| Проблема | Решение |
|---|---|
| Airflow не импортирует модули из `ingestion/` | Добавлен `sys.path.append` в DAG |
| PostgreSQL не создаёт таблицы | Переписан `db/init.sh` |
| Streamlit не обновляется после правок | Добавлен `--build` флаг к `docker compose up` |
| RapidAPI 429 (исчерпан месячный лимит) | Переход на Playwright + куки |
| Playwright не может зайти в Threads | Инжекция `sessionid` куки в браузерный контекст |
| RAG не находит совпадения | Снижен `similarity_threshold` до 0.5 |
| Скрапер статей падает по таймауту | Увеличен `execution_timeout` DAG до 6 часов |
